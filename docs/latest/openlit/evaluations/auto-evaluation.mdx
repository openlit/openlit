Auto Evaluation runs the [evaluation types](/latest/openlit/evaluations/overview) you've enabled against newly ingested LLM traces on a recurring schedule, without any manual "Run evaluation" click or SDK call. It's the production-monitoring counterpart to running evaluations manually or [programmatically via the SDK](/latest/sdk/quickstart-programmatic-evals).

## Manual vs. auto evaluation

| | Manual | Auto |
|---|---|---|
| Trigger | Click **Run evaluation** on a trace, or call the SDK | Cron schedule you configure |
| Scope | One trace at a time (or one SDK call) | All new LLM traces since the last run |
| Sampling | None - always runs | Configurable sample rate |
| Result tagging | `source: manual` | `source: auto` (or `AUTO_SKIPPED` if sampled out) |

## Enable auto evaluation

<Steps>
  <Step title="Open Evaluations settings">
    Navigate to **Evaluations > Settings** in the sidebar under the Configuration section.
  </Step>
  <Step title="Choose an evaluator model">
    Select the LLM provider, model, and Vault-stored API key that auto evaluation will use to judge responses.
  </Step>
  <Step title="Enable Auto Evaluation">
    Toggle the **Auto Evaluation** switch.
  </Step>
  <Step title="Set the sample rate">
    Enter the percentage of eligible traces to evaluate (e.g. `10` for 10%). Sampling is deterministic - the same trace always gets the same sampling decision - so a given trace can't be evaluated twice due to sampling changes between runs.
  </Step>
  <Step title="Set the cron schedule">
    Enter a standard cron expression, for example `*/15 * * * *` (every 15 minutes) or `0 * * * *` (every hour).
  </Step>
  <Step title="Save">
    OpenLIT installs a cron job that runs auto evaluation on the schedule you defined.
  </Step>
</Steps>

<Note>
Which evaluation types actually run (hallucination, bias, toxicity, or any custom type) is controlled separately on the **Evaluations > Settings > Evaluation Types** page, and can be linked to [Rule Engine](/latest/openlit/prompts-experiments/rule-engine) rules for conditional evaluation based on trace attributes like model, provider, or environment.
</Note>

## How auto evaluation selects and samples traces

On each cron tick:

| Step | What happens |
|------|-------------|
| **1. Incremental window** | Only traces ingested since the last successful cron run are considered. The first run processes all historical eligible traces. |
| **2. Sampling** | Each eligible trace is passed through a deterministic hash-based sample check against your configured sample rate. Traces that don't pass are recorded with `source: AUTO_SKIPPED` (skipped due to sampling, not an error). |
| **3. Evaluation types** | For sampled-in traces, every enabled evaluation type runs - using [Rule Engine](/latest/openlit/prompts-experiments/rule-engine) context as ground truth where a rule matches. |
| **4. Score and store** | Results are scored and stored with `source: auto`, visible in trace details and dashboards alongside manual evaluation results. |
| **5. Log run** | The cron log records totals for the run (evaluated, skipped, failed) so you can audit auto evaluation activity over time. |

### Cron restoration on restart

When the OpenLIT server restarts (e.g. after a container redeployment), all active auto-evaluation cron jobs are automatically restored from the database. No manual re-setup is needed.

## API reference

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/api/evaluation/config` | `GET` | Get the current evaluation config (auto flag, cron schedule, sample rate, evaluator model) |
| `/api/evaluation/config` | `POST` | Create or update evaluation config |
| `/api/evaluation/auto` | `POST` | Trigger an auto evaluation run (called by cron; allowlisted in middleware) |

## Configuration

The evaluation config is stored in the `EvaluationConfigs` table (managed by Prisma):

| Field | Type | Description |
|-------|------|-------------|
| `provider` / `model` | String | The LLM provider and model used to judge responses |
| `vaultId` | String | Vault secret used for the evaluator's API key |
| `auto` | Boolean | Whether auto evaluation is enabled |
| `recurringTime` | String | Cron schedule (e.g. `*/15 * * * *`) |
| `meta` | String (JSON) | Internal metadata including `evalSampleRate` and per-type `evaluationTypes` settings (enabled state + Rule Engine bindings) |
